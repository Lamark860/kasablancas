# Vlad rev1 — обзор проекта

Внутренний инструмент ландшафтного дизайнера для подбора растений клиенту через пересечение нескольких эзотерических/символических систем.

## Идея в одной фразе

"Вы вводите минимум данных о клиенте → мы прогоняем его через 5+ независимых систем (друиды, зодиак, имя, цвет глаз, стихия) → находим **пересечения** → получаем пул из 10-15 растений → эксперт фильтрует под участок → клиент получает персональную рекомендацию **без указания источников** = «уникальная авторская методика»."

## Что важно понимать

1. Это **не публичный сервис**, а **инструмент эксперта**. Публичная версия — этап 2.
2. Монетизация **не на инструменте**, а на услугах коллеги-ландшафтника, которые этот инструмент помогает продать.
3. Ценность для клиента = **эмоция** (магия персонализации) + **результат** (красивый сад).
4. Архитектура должна быть **расширяемой** — добавление новой "системы" не должно ломать ядро.

## Принцип "много слоёв = магия"

Если использовать только друидский календарь — клиент гуглит за 30 секунд. Если **5-7 систем + наша интерпретация** — проверить невозможно. Каждая система отдельно слаба, но **пересечения между ними** создают эффект "это про меня".

## Ключевые решения, которые надо принять

1. **MVP-набор систем** — предлагаю 5: друиды-деревья, друиды-цветы, зодиак, славянский, цвет глаз. Достаточно для пересечений; объём БД управляемый.

2. **Что хранить в БД растений** — есть в `09-architecture.md`. Главное: универсальные теги `element`, `gender_energy`, `planet`, `chakra` — они склеивают системы.

3. **Замены южных растений** — есть начальная таблица в `08-climate-zones.md`, нужно дозаполнить.

4. **Включаем ли натальную карту?** — В MVP **нет** (сложно для пользователя ввести время+место, и интерпретация требует астролога). На этапе 2-3 — да.

5. **Включаем ли модуль размещения (Багуа)?** — На этапе MVP+ (после первой версии). Это уже не "что посадить", а "куда" — отдельная ценность.

## Что предлагаю как следующий шаг

Когда обсудим и зафиксируем содержание:

**Вариант A:** Начать с **HTML-прототипа интерфейса** (форма ввода + макет результата). Без живых данных — на моках. Это даст быстрое визуальное понимание "как это будет выглядеть". Можно сделать сразу 2-3 варианта дизайна (например, "тёмный мистический", "светлый ботанический", "минималистичный").

**Вариант B:** Сначала **доделать данные** — собрать полные таблицы по 5 системам (~150-300 строк) и только потом интерфейс.

**Вариант C:** Сразу **рабочий прототип** с упрощёнными данными (по 5-10 строк на систему) — чтобы было живо и можно было прокликать. Дальше наращиваем БД.

Я бы шёл по варианту C — самый быстрый путь к "пощупать руками". Что думаешь?
