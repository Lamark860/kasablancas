# DECISIONS — архитектурные решения и компромиссы

Плоский справочник живых архитектурных решений. Хронология прогресса —
в `WORK_LOG.md`. Этот файл — для тех, кто хочет быстро понять «какие
короткие пути мы выбрали и что будет стоить от них откатиться».

Формат записи:

- **Решение** — что выбрали.
- **Почему** — мотивация / альтернативы / на чём срезали.
- **Цена возврата** — что нужно сделать, чтобы пересмотреть.
- **Где** — ключевые файлы.
- **Этап** — когда зафиксировано.

Каждое решение жывое, пока не помечено `~~отменено~~` с датой и причиной.

## 1. Порты 3100 / 8100 вместо 3000 / 8000

- **Решение:** хост-порты для api и web сдвинуты с дефолтных 3000/8000 на 3100/8100.
- **Почему:** на машине разработчика крутятся другие docker-проекты
  (`ai-generator-frontend` на 3000, `ai-generator-api` на 8000). Изменить
  свои порты дешевле, чем ломать соседей.
- **Цена возврата:** обратно на 3000/8000 — поменять три места
  (`docker-compose.yml`, `.env`, `.env.example`) и обновить `starter/README.md`.
- **Где:** `starter/docker-compose.yml`, `starter/.env(.example)`.
- **Этап:** 0.

## 2. Раздельный URL API для SSR и для браузера

- **Решение:** `NUXT_PUBLIC_API_BASE=http://localhost:8100` (для браузера) и
  `NUXT_API_BASE_SERVER=http://api:8000` (для SSR внутри docker-сети).
  Composable `useApi.ts` переключается через `import.meta.server`.
- **Почему:** из контейнера web `localhost` — это сам web, не хост, поэтому
  SSR-fetch на `localhost:8100` падал молча и страница рендерилась без ping.
  Альтернатива (host.docker.internal) хуже переносится на сервер.
- **Цена возврата:** при переезде в прод за reverse-proxy оба URL станут
  одним публичным — упростится до одного env.
- **Где:** `starter/web/nuxt.config.ts`, `starter/web/composables/useApi.ts`.
- **Этап:** 0.

## 3. Сидинг — ручной, не автоматический

- **Решение:** `python -m vlad.seed` запускается явно. CMD api-контейнера
  делает `alembic upgrade head` сам, но **не** сидит данные.
- **Почему:** roadmap говорит «reseed безболезненно» — команда должна
  очищать и заливать все эзотерические данные (oracle_entries, plants), не
  трогая Person и Recommendation. Если бы сид был автоматическим при
  каждом старте, ручные правки эксперта в `/admin` затирались бы при
  рестарте контейнера.
- **Цена возврата:** добавить `&& python -m vlad.seed` в CMD —
  одна строка. Но обратно — не вернуть, эксперт потеряет правки.
- **Где:** `starter/docker-compose.yml`, `starter/api/src/vlad/seed.py`.
- **Этап:** 1.

## 4. Plant.slug — стабильный ссылочный ключ (вместо FK на id)

- **Решение:** `oracle_entries.plant_slug` и `recommendations.title_plant_slug`
  ссылаются на `plants.slug` (уникальный текстовый), а не на `plants.id`.
- **Почему:** slug стабилен между миграциями БД и переписываниями JSON-сидов.
  Если переcоздать `plants` (что делает reseed на этапе 2 и далее), id'ы
  поменяются, а slug'и останутся. JSON-сиды соответствий ссылаются на slug —
  они не зависят от id-нумерации.
- **Цена возврата:** мигрировать FK на id — ничего не теряем, кроме
  человекочитаемости JSON. Полная autogenerate-миграция.
- **Где:** `starter/api/src/vlad/models/__init__.py`.
- **Этап:** 1.

## 5. Тесты используют `Base.metadata.create_all`, не Alembic

- **Решение:** в тестах схема создаётся через `Base.metadata.create_all` на
  in-memory SQLite. Alembic в тестах не гоняется.
- **Почему:** на порядок быстрее (~1с вместо ~10с на 80 тестах). Корректность
  миграций косвенно проверяется тем, что smoke-тест `/admin` использует
  реальную БД, накаченную через `alembic upgrade head` при старте контейнера.
- **Цена возврата:** добавить альтернативную фикстуру с пере-прогоном
  alembic — ~10 строк. Если миграции начнут расходиться с моделями —
  поймаем при первом сидинге, не критично.
- **Где:** `starter/api/tests/conftest.py`.
- **Этап:** 1.

- **Решение:** на 21 декабря оракул `druid_tree` возвращает `[]` —
  ни одно дерево не выпадает.
- **Почему:** в `research/02-druid-tree-calendar.md` Инжир покрывает
  12.12–20.12, Бук — только 22.12 (зимнее солнцестояние). 21.12 — лакуна

  расширить Бука до 21–22.12). Тест
  `test_each_date_resolves_to_exactly_one_tree` сразу укажет на расхождение.
- **Где:** `starter/api/data/seed/druid-tree.json`,
  `starter/api/tests/test_oracle_druid_tree.py`.
- **Этап:** 2.

## 7. 29.02 включено в период Сосны

- **Решение:** Сосна имеет интервал `02-19..02-29`.

  `research/02` указывает «19.02 – 28/29.02» — мы трактуем неоднозначно
  в пользу включения.
- **Цена возврата:** убрать `02-29` из end-границы — копеечно.
- **Где:** `starter/api/data/seed/druid-tree.json`.
- **Этап:** 2.

## 8. USDA-зоны для южных видов — приближённые

- **Решение:** для cypress, walnut, fig, olive, beech проставлены
  предварительные `min_zone_usda` (7, 6, 7, 8, 6 соответственно).
- **Почему:** эксперт ещё не зафиксировал канон, а без значения этап 6

- **Цена возврата:** UPDATE через `/admin/plant/edit` или правка
  `data/seed/plants.json` + reseed. Влияет только на фильтрацию по
  холодостойкости — на пул напрямую не влияет.
- **Где:** `starter/api/data/seed/plants.json`.
- **Этап:** 2.

## 10. `StaticPool` в фикстуре `db_session`

- **Решение:** in-memory SQLite в тестах создаётся с `poolclass=StaticPool`.
- **Почему:** TestClient FastAPI гоняет хэндлеры через threadpool (anyio).
  Без `StaticPool` каждая connection видит свою пустую in-memory БД, и
  роуты-тесты падают с `no such table: oracles`. С `StaticPool` все потоки
  делят одну connection.
- **Цена возврата:** перейти на файловый sqlite в тестах — медленнее, но
  без ловушки. Сейчас `StaticPool` достаточно.
- **Где:** `starter/api/tests/conftest.py`.
- **Этап:** 3.

## 11. Таблица `recommendations` пока не пишется

- **Решение:** `POST /recommend` считает пул, но **не сохраняет** результат
  в таблицу `recommendations`.
- **Почему:** история подборов и кураторская правка — фича этапа 9
  (полировка). На этапе 3 это лишний код, который пришлось бы потом
  переделывать под curated_pool / expert_notes.
- **Цена возврата:** добавить `db.add(Recommendation(...))` в роуте — десяток
  строк. Существующие рекомендации в БД не появятся (история начнётся с
  момента включения).
- **Где:** `starter/api/src/vlad/routes/recommend.py`.
- **Этап:** 3.

## 12. Moshier-эфемериды вместо файлов `.se1`

- **Решение:** `swe.calc_ut(..., swe.FLG_MOSEPH)` — встроенные в pyswisseph
  эфемериды Moshier, без скачивания `.se1` файлов.
- **Почему:** точность ~3 угловых секунды — с большим запасом для назначения
  знаков (зоны 30°). Файлы `.se1` ~10 МБ, требуют интернета при сборке и
  отдельной операции по складыванию в `api/ephe/`. Для нашего use case
  (sun_sign + element_count) — overkill.
- **Цена возврата:** скачать `seas_18.se1`, `semo_18.se1`, `sepl_18.se1` в
  `api/ephe/`, снять `FLG_MOSEPH`. Точность вырастет до 0.001″ — нужно для
  прогрессий, тесных аспектов, а не для нас.
- **Где:** `starter/api/src/vlad/natal/swisseph_wrapper.py` (`_FLAGS`).
- **Этап:** 4.

## 13. Без `birth_time` берём 12:00 UTC

- **Решение:** если пользователь не ввёл время рождения или таймзону, для
  расчёта используем 12:00 UTC того же дня.
- **Почему:** Sun и медленные планеты (Mars, Jupiter, Saturn, Uranus, Neptune,
  Pluto) от смещения внутри суток почти не зависят. Луна, Меркурий и Венера
  на стыке знаков могут ошибиться.
- **Цена возврата:** требовать `birth_time` обязательно — сильно повысит
  бариер ввода. Альтернативно: показывать в UI дисклеймер «для точного
  расчёта Луны желательно знать время».
- **Где:** `starter/api/src/vlad/natal/swisseph_wrapper.py` (`_to_utc`).
- **Этап:** 4.

## 14. Геокодинг — best-effort, не блокирующий

- **Решение:** при `POST /persons` с `birth_place` пробуем
  Nominatim+timezonefinder. При любой ошибке (нет сети, лимит, неизвестное
  место) Person сохраняется с пустыми `birth_lat/lon/tz`.
- **Почему:** иначе единая ошибка геокодера парализует весь intake. Эксперт
  может заполнить координаты руками через `/admin/person/edit`.
- **Цена возврата:** сделать строгий режим (вернуть 422 при сбое геокодера)
  — пара строк. Будет работать только в онлайне.
- **Где:** `starter/api/src/vlad/natal/geocode.py`,
  `starter/api/src/vlad/routes/persons.py`.
- **Этап:** 4.

## 15. Zodiac покрывает не все знаки и не все растения research/04

- **Решение:** в `data/seed/zodiac.json` 22 entries, охватывают 11 из 12
  знаков (Aries пуст). Цветы и травы (тюльпан, лаванда, роза, …) пока не
  сидируем.
- **Почему:** наш `plants.json` — 22 дерева/кустарника. Растения для Aries
  (барбарис, боярышник, лиственница, тюльпан, мак) и цветочные дополнения
  для других знаков туда не входят. Расширять `plants.json` ради zodiac —
  отдельная задача с участием эксперта.
- **Цена возврата:** при расширении `plants.json` добавить недостающие
  entries в `zodiac.json` — оракул не трогаем, тесты сами обновятся через
  `test_zodiac_seed_plant_slugs_exist`.
- **Где:** `starter/api/data/seed/plants.json`,
  `starter/api/data/seed/zodiac.json`.
- **Этап:** 4.

## 16. Eye_color и numerology покрывают только наши 22 растения

- **Решение:** в `data/seed/eye-color.json` и `data/seed/numerology.json`
  записаны только те mapping'и, чьи растения уже есть в `plants.json`. Для
  «зелёных», «чёрных» глаз и числа 7 (мистика) — только частичное покрытие
  (число 7 → cypress как замена тису/можжевельнику).
- **Почему:** расширять `plants.json` без участия эксперта (с ботаническими
  именами, USDA-зонами, hierarchy_potential) — заранее ставить чужой канон.
  Лучше иметь честный пробел, чем тихо подставлять некорректные данные.
- **Цена возврата:** добавить в `plants.json` папоротник, плющ, можжевельник,
  тис, бересклет, барбарис, золотарник и т.п., затем дополнить entries.
  Сами оракулы при этом не меняются.
- **Где:** `starter/api/data/seed/eye-color.json`,
  `starter/api/data/seed/numerology.json`,
  `starter/api/data/seed/plants.json`.
- **Этап:** 5.

## 17. Name-оракул — намеренно небольшая стартовая база

- **Решение:** в `data/seed/name.json` всего 9 entries: Ива, Иванна, Ясмина,
  Жасмин, Ясмин, Дарья (×2), Дарина, Ева. Сравнение case-insensitive по
  `first_name`.
- **Почему:** research/06 рекомендует подход «каждый клиент = повод добавить
  его имя в базу». Полная таблица имён православного календаря — отдельная
  работа на десятки часов и требует ботанической экспертизы. На MVP-этапе
  достаточно демонстрационной выборки.
- **Цена возврата:** просто дополнять `name.json`. Оракул и тесты при этом
  не меняются.
- **Где:** `starter/api/data/seed/name.json`,
  `starter/api/src/vlad/oracles/name.py`.
- **Этап:** 5.

## 18. Оракулы druid_flower, slavic, lunar — отложены

- **Решение:** на этапе 5 реализованы 5 из 8 оракулов в `oracles.json`.
  `druid_flower`, `slavic`, `lunar` оставлены с `implemented: false`.
- **Почему:**
  - **`druid_flower`** требует ~36 цветов/трав в `plants.json` (мак, лилия,
    наперстянка, ромашка, …). Без расширения справочника entries не на что
    ссылаться. Расширение справочника — отдельная работа с экспертом.
  - **`slavic`** (research/05) сам по источнику неполный — TODO «собрать
    25–30 периодов из одного источника». Реализовать без канона — заведомо
    ставить мусор.
  - **`lunar`** (research/07) — «источники не очень устойчивы», нет
    канонической таблицы лунный_день → растение. Кроме того, требует
    отдельного wrapper'а вокруг pyswisseph для расчёта лунного дня.
- **Цена возврата:** все три оракула устроены так же, как уже сделанные —
  один файл-сид + класс-наследник Oracle + регистрация. ~2-3 часа на каждый
  при наличии данных. Архитектурного блока нет, чисто контентный.
- **Где:** не реализовано; место — `starter/api/src/vlad/oracles/`.
- **Этап:** 5.

## Приложение: как добавлять новые записи

Когда фиксируем новый компромисс:

1. Запись здесь — с цифровым id (продолжение нумерации).
2. В `WORK_LOG.md` для соответствующего этапа в разделе «Заметки и
   компромиссы» — короткая ссылка вида `См. DECISIONS.md §16.`
3. Если решение позже отменяется — не удаляем, а помечаем
   `~~отменено DD.MM.YYYY: причина~~` и оставляем как след истории.
