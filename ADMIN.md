# Hortus Animæ — гайд по админке

Это инструкция для эксперта, который ведёт каталог растений, оракулы и соответствия. Предполагается, что проект уже запущен по [START.txt](START.txt) (Docker Desktop + `start.bat`/`start.command`).

> Если ты разработчик и хочешь поправить логику — это не сюда; см. `handoff/` в корне.

---

## 1. Где админка

Когда проект поднят, открой в браузере:

```
http://localhost:8100/admin
```

Если порт занят и в `starter/docker-compose.yml` ты поменял `8100` на другое число — подставь его. Авторизации нет: админка живёт локально и в публичную сеть не торчит.

---

## 2. Что внутри: карта разделов

В левом меню админки восемь пунктов. Кратко:

| Раздел           | Что это                                                              | Кто правит                     |
| ---------------- | -------------------------------------------------------------------- | ------------------------------ |
| **Клиенты**      | Гостьи, по которым делается подбор                                   | в основном создаются через UI  |
| **Растения**     | Каталог: имя, зона, свет, символика                                  | эксперт                        |
| **Оракулы**      | «Источники» подбора (зодиак, имя, цвет глаз и т.д.)                  | эксперт, редко                 |
| **Соответствия** | Связи «оракул → растение» с условием (когда срабатывает и с каким весом) | эксперт, постоянно         |
| **Рекомендации** | История подборов (read-only, не трогать)                             | автоматически                  |
| **Натальные карты** | Кэш расчётов (read-only)                                          | автоматически                  |
| **Импорт CSV**   | Массовое добавление соответствий из таблицы                          | эксперт                        |
| **Выгрузить в JSON** | Сохранить текущее состояние БД в `data/seed/*.json` (чтобы закоммитить в git) | эксперт      |

Дальше — по каждому разделу подробно.

## 3. Растения (Plants)

Это справочник всех растений, которые система знает. У каждого растения уникальный английский ключ-`slug` (`oak`, `willow`, `cypress`), к которому привязаны все оракулы.

### Какие поля заполнять

Обязательные:

- **slug** — латиницей, без пробелов: `oak`, `birch_silver`. Менять у уже привязанного растения нельзя — потеряются связи с оракулами.
- **name_ru** — русское имя, видит клиент в отчёте: `Дуб`, `Берёза серебристая`.
- **category** — выбери из: `tree`, `shrub`, `perennial`, `annual`, `grass`, `fern`, `vine`, `water`, `bulb`, `succulent`, `herb`.

Сильно влияют на подбор:

- **name_lat** — латинское имя для PDF (`Quercus robur`). Курсивом под именем.
- **min_zone_usda** — минимальная USDA-зона. Если у клиента сад зоны 4, а здесь стоит 7 — растение отфильтруется в режиме «не зимует».
- **max_zone_usda** — обычно пусто. Заполняй только если южные регионы реально не подходят.
- **sun** — освещение: `sun`, `part_shade`, `shade`, `sun_or_part_shade`, `any`. Должно сходиться с садом клиента.
- **soil_moisture** — `dry`, `normal`, `wet`, `any`.
- **hierarchy_potential** — 1..5. Насколько растение «достойно» быть **главным деревом**. 5 — дуб, ива, яблоня; 1 — мелкая трава.
- **is_weed_like** — `1` если склонно к экспансии (полынь, мята). Не выкидывает, но в подборе вес ×0.5.

Опциональные (попадают в PDF):

- **short_story** — 1–2 предложения. Появляется цитатой под именем растения в клиентском листе.
- **long_story** — длинное описание, на будущее (пока не выводится).
- **image_url** — задел под иконки.

Символика (на будущее):

- **element** — `огонь / вода / земля / воздух`.
- **gender_energy** — `masculine` / `feminine` / `neutral`.
- **planet**, **chakra** (1..7), **bloom_months**, **bloom_color**, **autumn_color**, **evergreen**, **shelter_friendly**, **height_max_m**, **width_max_m**, **growth_speed** — пока не используются в подборе, но индексируются. Можно заполнять впрок.

### Куда попадает результат

- `name_ru` + `name_lat` + `short_story` → в карточку оракула на странице эксперта и в клиентский PDF.
- `min_zone_usda`, `sun`, `soil_moisture` → в фильтры эксперта (галочка «не зимует» / «свет/почва»).
- `hierarchy_potential` → сортировка пула, главное дерево выбирается с учётом этого числа.
- `is_weed_like` → флажок «приглушить сорные» × 0.5.

## 4. Оракулы (Oracles)

Оракул = источник, который сопоставляет данные клиента (день рождения, имя, цвет глаз и т.д.) с растениями.

В админке менять у оракула почти ничего не нужно:

- **active** — `1` если оракул участвует в подборе, `0` чтобы временно выключить. Эксперт на странице `/expert/{id}` тоже может включать/выключать оракулы для конкретной гостьи, но это разовое — глобальный флаг `active` важнее.
- **weight** — глобальный множитель вклада оракула. Стандарт `1.0`. Поставь `0.5`, чтобы голос этого оракула значил меньше; `1.5` — больше.
- **sort_order** — порядок в списке источников у эксперта и в клиентском PDF. Меньше число → выше.
- **required_inputs** — JSON-список полей `Person`, без которых оракул не запустится. Обычно стоит правильное значение из seed-файлов, трогать не нужно.

> Создавать **новые оракулы** через админку нельзя: для оракула должен существовать класс-наследник `Oracle` в коде. Если нужен новый — пиши разработчику.

## 5. Соответствия (OracleEntries) — самая горячая таблица

Это правила вида **«если у клиента _X_, то предложи _растение Y_ с весом _w_»**. Их сотни.

### Поля

- **Оракул** (`oracle_id`) — выбери из списка.
- **Растение** (`plant_slug`) — выбери из списка.
- **Тип matcher** — что именно проверять у гостьи. Пять вариантов:
  - **Период (даты от/до, ежегодно)** — `matcher_from`/`matcher_to` в формате `MM-DD`, например `03-05` … `04-01`. Период может оборачиваться через Новый год: `12-23` … `01-01`.
  - **Знак зодиака** — `matcher_sign`: `aries`, `taurus`, …, `pisces` (12 знаков).
  - **Число имени** (Пифагор) — `matcher_number`: 1..9.
  - **Цвет глаз** — `matcher_color`: `blue`, `grey`, `green`, `hazel`, `brown`, `dark`, `amber`.
  - **Имя клиента** — `matcher_name`: текст, сравнение без учёта регистра.
- **weight** — вес одного голоса. `main` обычно `1.0`, `companion` — `0.7..0.9`.
- **role** — `main` (главное дерево) или `companion` (спутник). Пусто = просто запись в пуле.
- **reason_for_expert** — короткий текст, который видит только эксперт в `/expert/{id}`. Удобно держать «почему» — это помогает обосновать подбор.
- **reason_for_client** — 1 предложение, попадает в клиентский лист и PDF.
- **sort_order** — порядок внутри записей одного оракула. Обычно `0`.

### Защита от дублей

Если ты попытаешься создать запись с тем же `(оракул, растение, matcher)`, что уже есть — админка скажет «такое соответствие уже существует, отредактируй существующее».

## 6. Импорт CSV — массовое добавление соответствий

Если соответствий много (сотни строк), их удобнее залить таблицей.

Раздел в меню — **«Импорт CSV»**.

### Как это работает

1. Сверху выбери шаблон — например `druid_tree`, `zodiac`, `numerology`, `eye_color`, `name`, или `все типы`. Скачается CSV-заготовка с правильными колонками.
2. Открой её в Excel/Numbers/LibreOffice, отредактируй строки.
3. Скопируй всю таблицу обратно в текстовое поле «Импорт CSV» и нажми **«Просмотреть изменения»**.
4. Админка покажет таблицу с пометками:
   - <span style="color:#2da44e">+ новая</span> — будет создана.
   - <span style="color:#bf8700">≈ обновится</span> — найдена существующая по `(oracle_id, plant_slug, matcher)`, поля поменяются.
   - = без изменений — совпадение, пропустим.
   - <span style="color:#d4380d">× ошибка</span> — что-то не так со строкой (например, нет такого slug растения). Детали справа.
5. Если ошибок нет — нажми **«Применить»**. Если есть — поправь CSV и повтори.

### Формат CSV

Колонки (точные имена в заголовке):

```
oracle_id,plant_slug,matcher_type,matcher_from,matcher_to,matcher_sign,matcher_number,matcher_color,matcher_name,weight,role,reason_for_expert,reason_for_client,sort_order
```

В каждой строке заполнены только те `matcher_*`, которые относятся к выбранному `matcher_type`. Остальные — пусто.

Пример (друидский календарь, дуб на летнее солнцестояние):
```
druid_tree,oak,date_range_yearly,06-21,06-21,,,,,1.0,main,"летнее солнцестояние — день дуба","ты родилась в день дуба",0
```

Пример (зодиак, ива для Рака):
```
zodiac,willow,zodiac_sign,,,cancer,,,,1.0,main,"Луна — стихия Рака","ива дружит со знаком Рака",0
```

Запятые внутри текста — оборачивай поле в двойные кавычки `"..."` как обычно.

## 7. Сохранить изменения в репозиторий (Выгрузка JSON)

База `data/local.db` живёт в Docker-volume и **не коммитится в git**. Чтобы твои правки увидели остальные (и чтобы они пережили `docker compose down -v`) — нажми в меню админки **«Выгрузить в JSON»** → **«Выгрузить сейчас»**.

Что произойдёт:
- админка сериализует таблицы `plants`, `oracles`, `oracle_entries` в `starter/api/data/seed/*.json`,
- покажет сводку: сколько записей в каждом файле.

Потом, чтобы зафиксировать изменения:

```bash
git add starter/api/data/seed/
git commit -m "правки каталога: добавлен дуб, обновлены друиды"
git push
```

Коллега, который сделает `git pull` и заново поднимет проект, увидит твои данные — при пустой БД seed-файлы загружаются автоматически на старте.

> **Что не выгружается:** клиенты (`Person`), их рекомендации и натальные карты — это пользовательские данные, не справочник, и в git им не место.

## 8. Чего НЕ трогать в админке

- **Рекомендации** — это снимок подбора эксперта для конкретной гостьи, версионируется. Удалять / редактировать вручную нельзя: сломаются ссылки `/leaf/{token}` и история на странице эксперта.
- **Натальные карты** — кэш расчётов (Sun/Moon/ASC). Пересчитываются автоматически.

Если очень надо что-то снести — лучше удалить саму гостью (`Клиенты` → удалить), её рекомендации и натальную карту удалятся каскадом.

---

## 9. Где увидеть результат правок

Изменения в админке применяются сразу, перезапускать ничего не нужно.

- **Растения / Соответствия / Оракулы** → откроется `/expert/{id}` любой гостьи и нажмёшь «свести источники заново».
- **short_story / name_lat** → видно в клиентском листе `/leaf/{token}` и в PDF (`http://localhost:8100/reports/{person_id}.pdf`).
- **active у оракула** → отразится в колонке fontes на странице эксперта (он либо доступен галочкой, либо выключен в БД).

## 10. Шпаргалка: частые задачи

**Добавить новое растение:**
1. `Растения` → «Создать».
2. slug (англ., уникальный) + name_ru + category. Минимум.
3. Если потребуется в подборе → потом добавь хотя бы одно соответствие на нужный оракул.

**Привязать «дуб» к знаку зодиака Лев:**
1. `Соответствия` → «Создать».
2. Оракул = `zodiac`, Растение = `oak`, тип = `Знак зодиака`, знак = `leo`, role = `main`, weight = `1.0`, текст-причины — на твой вкус.

**Поменять имя растения:**
1. `Растения` → выбери строку → исправь `name_ru` / `name_lat` → «Сохранить». slug не трогай.

**Временно отключить оракул:**
1. `Оракулы` → строка → `active = 0` → «Сохранить».

**Откатить всё из git-seed:**
1. Останови проект (`stop.command` / `stop.bat`).
2. Удали `starter/api/data/local.db` (или просто volume через Docker Desktop).
3. Запусти заново — seed-файлы из репозитория накатятся как новая БД.

## Если что-то сломалось

- Админка не открывается → проверь что Docker Desktop запущен и в `Containers` зелёные `vlad-api` и `vlad-web`.
- В `Соответствия` ругается «дубль» → значит такая комбинация (оракул, растение, matcher) уже есть; найди существующую запись и отредактируй её.
- После «Применить» в CSV-импорте ничего не изменилось → скорее всего все строки в статусе `=` (без изменений). Проверь, что у CSV-строк действительно другие значения.
- Импорт CSV падает с «ошибка парсинга» → открой CSV в текстовом редакторе, проверь что разделитель — запятая, кавычки парные, кириллица в UTF-8.

Если совсем тупик — пиши в чат проекту.
