# Стек

## Итоговый выбор

```
Frontend:    Nuxt 3 (Vue 3, TypeScript, SSR опционально)
Backend:     FastAPI (Python 3.11+) + SQLAlchemy 2.0 + Alembic
БД:          SQLite (этап 1, локально) → PostgreSQL (этап 2, на хостинге)
Натальная:   pyswisseph (Python-обёртка над Swiss Ephemeris)
Админка БД:  sqladmin (FastAPI-нативная админка) ИЛИ Adminer (отдельный контейнер)
PDF:         WeasyPrint (HTML → PDF, кириллица из коробки)
Валидация:   Pydantic 2 на бэке, Zod (или просто TS-типы) на фронте
Менеджеры:   uv (Python), pnpm (Node) — оба самые быстрые
Контейнер:   Docker Compose для dev и prod (api + web + postgres + adminer)
```

### Почему так

1. **FastAPI + Pydantic** — идеально под "много разных оракулов с типизированным выводом". Каждый оракул возвращает `OracleResult` (Pydantic-модель), и всё автоматически валидируется и документируется в OpenAPI.

2. **SQLAlchemy 2.0** (а не Tortoise/Peewee) — самый зрелый ORM для Python, отлично работает с Alembic-миграциями, легко переезжает SQLite → Postgres сменой DSN.

3. **pyswisseph** — нативная Python-обёртка над Swiss Ephemeris, тот же движок, что используют все астрологические сервисы. Бесплатно, офлайн, точно.

4. **Nuxt 3 vs чистый Vue + Vite** — Nuxt даёт из коробки роутинг, авто-импорты, SSR (если захочешь индексироваться поисковиками когда выйдешь в паблик), серверные API-роуты (можно даже без отдельного FastAPI обойтись — но мы не будем). Для одного разработчика Nuxt дешевле в поддержке.

5. **SQLite → Postgres** — на этапе 1 это **один файл** на твоём ноуте. Скопировал — забэкапил. Когда понадобится бот / публичный сервис — меняешь `DATABASE_URL` в `.env`, прогоняешь миграции, переносишь данные. SQLAlchemy не заметит подмены.

6. **WeasyPrint, не Playwright** — для Python-стека логичнее. Берёт HTML+CSS, выдаёт PDF. Кириллица, шрифты с засечками, фоновые текстуры — всё работает. Playwright тоже можно, но он тащит браузер в зависимости.

7. **sqladmin** — встраивается прямо в FastAPI на отдельном пути (`/admin`), даёт CRUD для твоих SQLAlchemy-моделей. Альтернатива — поднять Adminer контейнером, но это лишняя движуха для MVP.

## Где хостить

Я бы рекомендовал такую последовательность:

### Этап 1 (сейчас) — локально
Никакого хостинга. Запускаешь `docker compose up` на своём ноуте, открываешь `localhost:3000` (фронт) и `localhost:8000/admin` (админка). Этого достаточно — вы с коллегой работаете оба со своих машин, БД синхронизируете через git+SQLite-файл (или Dropbox).

### Этап 2 (внутренний доступ для двоих) — VPS в РФ
**Selectel** или **Timeweb Cloud** — самые надёжные, есть нормальная панель, оплата картой РФ, серверы в РФ (важно для скорости и юрисдикции). Минимальная конфигурация:
- 2 vCPU, 4 GB RAM, 40 GB SSD
- Ubuntu 22.04
- Docker Compose с тем же стеком + nginx + Let's Encrypt
- Цена: ~600-1000 ₽/мес

Закрываешь basic-auth или TG-логином — это всё ещё внутренний инструмент, не паблик.

### Этап 3 (публичный бот + сайт)
Тот же VPS, либо переезд на что-то посерьёзнее (Yandex Cloud / managed Postgres). К тому моменту будет понятно по нагрузке.

**Не рекомендую** для этого проекта: Vercel, Railway, Heroku — они либо отрубают РФ-карты, либо имеют ограничения на длительные процессы (а WeasyPrint и pyswisseph любят память).

## Что не берём и почему

| Не берём | Почему |
|----------|--------|
| Django | Избыточен для API-only бэка. FastAPI быстрее и легче. |
| Flask | Устаревший дефолт. FastAPI — современная замена. |
| Vue 2 / options API | EOL. Только Composition API на Vue 3. |
| Tailwind | Эстетика проекта (викторианский гримуар) лучше пишется обычным CSS. Tailwind хорош для типового UI, тут — нетипичный. |
| MongoDB | Данные строго табличные. SQL подходит идеально. |
| Внешние астро-API | Платно, зависимость от чужого сервиса, оффлайн не работает. pyswisseph закрывает 100% задач. |
